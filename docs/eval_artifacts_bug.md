# Eval bug — `artifacts/` path mismatch caps Mode 1 at score=1

**Status:** open
**Found:** 2026-05-25 during the Mode 1 (`--mode env`) benchmark of `r2r++` vs
`causal-mem` vs `kernel-agent` on the 10-paper kernel subset
(`benchmark/harness/cache/tasks_kernel_subset.json`).
**Branches affected:** all three (`r2r++`, `causal-mem`, `kernel-agent`).
`benchmark/harness/runner_repo2rocm.py` is content-identical across all
worktrees, so the same patch applies to each.

## TL;DR

Every Repo2ROCm task in this benchmark is scoring **1/5 ("loop_complete")**
regardless of branch, even when the inner agent visibly emits
`ROCM_ENV_VERIFIED` and the run.log prints "Generate success!". The cause is
a **CWD / relative-path mismatch** between the harness and the build_agent
subprocess: the harness writes outputs to one location and looks for them
in another, so `metadata.json` always reports `copied_artifacts: []`. With
`artifacts/` empty, `extract.py` cannot find any markers, so the rubric
defaults `rocm_env_verified=False`, `dockerfile_present=False`,
`build_success=False` → the rubric returns the loop-complete floor of 1/5.

This is **not** a per-track regression. It is a harness instrumentation bug
that masks any improvement Track 1 (causal memory) or Track 2 (kernel
converter agent) could deliver. **The Mode 1 comparison cannot be
interpreted until this bug is fixed.**

## Symptom

On a clean run of all three branches against the 10-paper Mode 1 subset:

| Branch | n | rocm_env_verified_rate | score>=3 rate | score==1 rate |
|---|--:|--:|--:|--:|
| r2r++ (baseline) | 10 | 0 % | 0 % | 100 % |
| causal-mem | 10 | 0 % | 0 % | 100 % |
| kernel-agent | 10 | 0 % | 0 % | 100 % |

Source: `<worktree>/benchmark/reports_mode1/results.csv` and `summary.md`
generated by `python -m harness.main report`.

the rubric records `rocm_env_verified=False, score=1`.

## Root cause

`benchmark/harness/runner_repo2rocm.py` constructs `root_path` as a
relative path and passes it that way to the build_agent subprocess:

```114:130:benchmark/harness/runner_repo2rocm.py
    # Per-task root directory keeps each Repo2ROCm run's `output/`, `kb/`,
    # and `utils/repo/` isolated so parallel runs do not collide.
    root_path = os.path.join(task_dir, "root")
    os.makedirs(root_path, exist_ok=True)

    api_key = os.environ.get("AMD_LLM_API_KEY", "")
    cmd = [
        "python3", "-u", os.path.join(REPO_ROOT, "build_agent", "main.py"),
        "--full_name", full_name,
        "--sha", sha,
        "--root_path", root_path,
        ...
    ]
```

The harness then launches the build_agent with `cwd=REPO_ROOT`:

```184:192:benchmark/harness/runner_repo2rocm.py
    with open(log_path, "wb") as logf:
        proc = subprocess.Popen(
            cmd,
            cwd=REPO_ROOT,
            stdout=logf,
            stderr=subprocess.STDOUT,
            env=env,
            preexec_fn=os.setsid,
        )
```

After the subprocess returns, the harness copies a fixed set of artifacts
from `<root_path>/output/<full_name>/` into `<task_dir>/artifacts/`,
*still using the relative `root_path`*:

```202:220:benchmark/harness/runner_repo2rocm.py
    artifacts_root = os.path.join(task_dir, "artifacts")
    out_repo_dir = os.path.join(root_path, "output", full_name)
    copied = []
    for fname in (
        "test.txt", "track.json", "track.txt",
        "outer_commands.json", "inner_commands.json",
        "paper_reproduction.json", "plan.txt", "Dockerfile",
        "agent_debug_log.txt", "sha.txt",
    ):
        src = os.path.join(out_repo_dir, fname)
        dst = os.path.join(artifacts_root, fname)
        if _copy_if_exists(src, dst):
            copied.append(fname)
```

The two processes have **different cwds**:

| Process | cwd | Resolves `runs_mode1/<task>/repo2rocm/root` to |
|---|---|---|
| build_agent subprocess | `REPO_ROOT` (e.g. `/home/upandey/rocm-causal-mem`) | `/home/upandey/rocm-causal-mem/runs_mode1/<task>/repo2rocm/root` |
| harness (after subprocess returns) | wherever the launcher cd'd to (e.g. `/home/upandey/rocm-causal-mem/benchmark`) | `/home/upandey/rocm-causal-mem/benchmark/runs_mode1/<task>/repo2rocm/root` |

So the build_agent writes its outputs to one absolute path and the
harness `_copy_if_exists` checks a different absolute path. Every check
fails silently (the helper returns False on missing source), `copied`
stays `[]`, `metadata.json["copied_artifacts"]` is empty, and the
`artifacts/` dir is created but never populated.

## Evidence

1. **`metadata.json` shows empty `copied_artifacts` for every completed
   task across all three branches.** Example
   (`/home/upandey/rocm-causal-mem/benchmark/runs_mode1/126-stacey-promoting-stochastic-steepest-descent/repo2rocm/metadata.json`):

   ```json
   {
     "paper_id": "126-stacey-promoting-stochastic-steepest-descent",
     "approach": "repo2rocm",
     "mode": "env",
     "cmd": [
       "python3", "-u",
       "/home/upandey/rocm-causal-mem/build_agent/main.py",
       ...
       "--root_path",
       "runs_mode1/126-stacey-promoting-stochastic-steepest-descent/repo2rocm/root",
       "--mode", "env",
       ...
     ],
     "exit_code": 0,
     "timed_out": false,
     "copied_artifacts": []
   }
   ```

   Note `--root_path` is the relative path. `copied_artifacts` is `[]`
   despite `exit_code=0`.

2. **The source files DO exist** — at the build_agent's location, not the
   harness's. For the same Stacey task:

   ```
   /home/upandey/rocm-causal-mem/runs_mode1/126-stacey-promoting-stochastic-steepest-descent/repo2rocm/root/output/xinyuluo8561/Stacey/track.json
   /home/upandey/rocm-causal-mem/runs_mode1/126-stacey-promoting-stochastic-steepest-descent/repo2rocm/root/output/xinyuluo8561/Stacey/inner_commands.json
   /home/upandey/rocm-causal-mem/runs_mode1/126-stacey-promoting-stochastic-steepest-descent/repo2rocm/root/output/xinyuluo8561/Stacey/outer_commands.json
   ```

   The harness was looking under
   `/home/upandey/rocm-causal-mem/benchmark/runs_mode1/...` — the
   `benchmark/` segment is what diverges.

3. **`track.json` contains the `ROCM_ENV_VERIFIED` literal**. The build_agent
   conversation log contains the string 11 times. `extract.py`'s detection
   logic is:

   ```131:131:benchmark/harness/scoring/extract.py
       rocm_verified = "ROCM_ENV_VERIFIED" in test_txt or "ROCM_ENV_VERIFIED" in track_blob
   ```

   `track_blob = _track_text(track_json)`. With the source `track.json`
   never copied to `artifacts/track.json`, `track_blob` is `""`, the
   `in` test fails, and `rocm_verified=False`.

4. **The rubric correctly cascades to 1/5** given the missing markers
   (`benchmark/harness/scoring/rubric.py:94-122`):

   ```text
   rocm_verified=False → not 3/4/5
   build_ok=False     → not 2
   exit_code=0 (≠127) → 1, "loop_complete"
   ```

   So the score=1 result is the **expected output of the rubric given the
   inputs it receives** — the bug is upstream of the rubric, at the copy step.

5. **The runner does not gate the copy on mode.** Lines 205-216 are
   unconditional; the same files are attempted for `env`, `reproduce`,
   and `full`. So this is not "Mode 1 is by design score-capped at 1" —
   the architecture supports `score>=3` in env-mode if the copy succeeds.

## Impact on this benchmark

Because every task on every branch is force-scored to 1/5, the comparison
that this benchmark was designed to make — "does causal memory or the
kernel converter agent move `rocm_env_verified_rate`?" — is **structurally
unanswerable** from this run's CSVs.

What we *can* tell from the data we have, despite the bug:

- All 30 tasks (10 papers × 3 branches) reached `exit_code=0` and emitted
  the build_agent's own success banners on the same 8 of 10 papers
  (controls: 104, 114, 126, 138, 20, 44, 55, 7).
- `30-llm-reproducibility` failed identically across all three branches
  with a pexpect EOF inside the harness's docker exec — harness-side,
  not track-attributable.
- `183-GCTM` crashed on `r2r++` with `rich.errors.MarkupError` (a logging
  rendering bug, not migration logic). Both variants completed the loop;
  causal-mem rescued the run-log completion specifically.
- Causal-mem's per-turn `[CAUSAL]` injection works (1,928 `[CAUSAL]`
  lines across 9 run.logs in causal-mem, 0 in baseline) and distillation
  works (5 → 6 transitions on at least one task), so the variant
  machinery is functional — we just can't measure its effect on the
  rubric yet.

## Proposed fix

Make `root_path` absolute at the function entry, so both the build_agent
subprocess invocation and the post-run copy step resolve to the same
location regardless of the harness's cwd. One-line change in
`benchmark/harness/runner_repo2rocm.py`:

```diff
--- a/benchmark/harness/runner_repo2rocm.py
+++ b/benchmark/harness/runner_repo2rocm.py
@@ -114,7 +114,7 @@
     # Per-task root directory keeps each Repo2ROCm run's `output/`, `kb/`,
     # and `utils/repo/` isolated so parallel runs do not collide.
-    root_path = os.path.join(task_dir, "root")
+    root_path = os.path.abspath(os.path.join(task_dir, "root"))
     os.makedirs(root_path, exist_ok=True)
```

Effects:

- The build_agent receives an absolute `--root_path`, so it writes outputs
  to the same absolute location regardless of its own cwd handling.
- The post-run `out_repo_dir = os.path.join(root_path, "output", full_name)`
  also becomes absolute, so `_copy_if_exists` checks the right place.
- Outputs move from `<repo>/runs_mode1/...` (where they used to land) to
  `<repo>/benchmark/runs_mode1/.../root/output/<full_name>/` — which is
  the sibling of `metadata.json`. Consistent with the rest of the harness.
- No behavioral change for any other call site; `root_path` is only
  consumed via the cmd and the copy step.

This is also a `causal-mem` / `kernel-agent` change because the file is
content-identical across the three worktrees. Apply verbatim to each.

## Related instrumentation gaps (out of scope for this fix, but worth flagging)

These are real and worth tracking even after the copy fix lands:

- **`paper_reproduction.json` is only written in paper-reproduction mode.**
  `build_agent/main.py:672` writes it inside a verdict/repro branch that
  doesn't fire under `--mode env`. After the copy fix, `track.json` is
  enough for `score=3` (ROCM_ENV_VERIFIED markers are extracted from
  `track_blob`), but `score=4/5` and the `success_report.*` columns in the
  CSV will still be 0 until a minimal env-mode success_report is also
  written. Suggested follow-up: write a slim `paper_reproduction.json`
  with just `{"verdict": "env_verified", "rocm_env_verified": True,
  "success_report": {"env": {"score": X}}}` at end of env-mode runs.
- **Token usage is 0 in every CSV row.** `_sum_outer_commands_usage` reads
  `outer_commands.json`, which lives in the same place as `track.json`.
  After the copy fix this should populate automatically; verify in the
  re-run.
- **Causal-mem's per-turn `[CAUSAL]` injection has no relevance gate.**
  Stacey received 3 seeded transitions about flash-attn / hipify-clang /
  paper-metric-mismatch on every observation turn despite Stacey being a
  pure-PyTorch repo with none of those issues. Add a `min_similarity` or
  `error_class match required` gate in
  `BuildMemoryProvider._provide_causal_memory_per_turn` so causal lines
  only fire when the situation matches. Otherwise the variant pays a
  ~2–10× token premium for noise prompts on simple repos.
- **GPU passthrough vs `gfx950`.** On the test host the build_agent's
  `torch.cuda.is_available()` returns False inside `rocm/pytorch:7.2.3`
  containers despite the host having 8 functional MI355X (gfx950) GPUs.
  The agent eventually tries `HSA_OVERRIDE_GFX_VERSION=9.4.2` but
  inconsistently. After the copy fix some tasks may still legitimately
  fail to reach `score=3` for this reason; that is a real result, not an
  instrumentation artifact. Recommend setting
  `HSA_OVERRIDE_GFX_VERSION=9.4` (or rebuilding the image with gfx950
  support) at container-launch time in the harness so the agent starts
  from a working baseline.

The first three follow-ups are agent-internal; the fourth is
host/infra-internal. None block the copy fix.
